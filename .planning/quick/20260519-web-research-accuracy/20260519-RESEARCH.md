# Web Research: External Techniques for GPS Coordinate Accuracy

**Researched:** 2026-05-19
**Domain:** Assignment algorithms / georeferencing / arc-length placement / map matching
**Confidence:** HIGH (core algorithms), MEDIUM (applicability to this exact domain)

---

## 1. Summary of Findings

### What the internal research already established

The root cause of the 36 m error on Joao Born is a **pole mis-assignment** problem, not a
calibration math bug. Page 3 has 144 Poste symbols vs. 14 actual route posts (10:1 false
candidate ratio). The beam-search (N3) did not close the gap. A full affine fit on the
mis-assigned positions already has a 5.13 m RMS residual floor — meaning no amount of
calibration layer tuning will reach < 5 m until the *input positions* are correct.

The canonical fix is N1: walk Cabo Projetado by arc-length using Distancia_Poste labels.
This is implemented but off by default (opt-in via `enableCableArcPlacer`). All of N2–N7
represent calibration-layer improvements that cannot beat the 5 m input-noise floor.

### What this web research adds

Four independent bodies of literature converge on the same conclusion and provide
concrete algorithmic tools:

1. **HMM / Viterbi map-matching** (Newson & Krumm 2009, Leuven Map Matching) — the
   standard formulation for "assign observations to a path given noisy positions" is
   directly applicable to the pole-assignment problem. It replaces greedy nearest-neighbor
   with a globally optimal DP that respects sequence order and inter-point distance
   constraints. O(n x k) time where n = posts, k = candidates per post.

2. **Arc-length parameterization + distance-label placement** — placing points on a curve
   given cumulative inter-point distances is a closed-form walk on a polyline (no
   optimization needed). This is exactly N1. The SVG `getPointAtLength()` API and the
   `point-at-length` npm package confirm the pattern is standard and browser-native.

3. **Hungarian / Munkres assignment** (munkres-js, munkres-algorithm npm) — for cases
   where you *do* need a global matching without sequence order (e.g., cross-page tap pole
   assignment), O(n^3) bipartite matching is available in small, dependency-free browser
   packages.

4. **PDF coordinate accuracy** — confirmed that PDF uses single-precision float (~7
   significant decimal digits). For a page of ~1200 pt width representing ~300 m, 1 PDF
   point = 0.36 m, so the float precision floor is < 0.01 m — not a limiting factor.
   The real limiting factor is coordinate system misidentification (CTM matrix errors)
   and pole-symbol mis-assignment, both of which are input-layer bugs.

**Primary recommendation:** Enable N1 (cable arc-length walk) as the default placement
strategy, using the Viterbi/HMM formulation (see section 3) as the improved fallback when
arc-length data is sparse. The Hungarian algorithm handles the residual tap-pole assignment.

## 2. Best External Techniques Applicable to This Problem

### 2.1 Assignment Algorithms

#### 2.1.1 HMM + Viterbi Decoding (MAP estimate over ordered sequence)

**Source:** Newson & Krumm 2009 — "Hidden Markov Map Matching Through Noise and Sparseness"
[CITED: microsoft.com/en-us/research/publication/hidden-markov-map-matching-noise-sparseness/]

This is the canonical algorithm for the problem of "given a sequence of N observations (GPS
points, noisy and sparse), assign each one to the most likely position on a road network
(polyline)." The analogy to pole assignment is exact:

| Map matching domain | Pole assignment domain |
|---------------------|------------------------|
| GPS observation at time t | Numero_Poste label anchor (anchorX, anchorY) |
| Road segments (candidates) | Poste symbols near the cable |
| Emission probability | Gaussian(dist(label, symbol), sigma=20 pt) |
| Transition probability | exp(-|arc_dist - label_m| / beta) |
| Viterbi output | Globally optimal symbol assignment |

**Emission probability** (how likely is symbol s to correspond to post i):

```
p_emit(i, s) = (1 / (sqrt(2*pi)*sigma)) * exp(-d(label_i, s)^2 / (2*sigma^2))
```

where `d` is Euclidean distance in PDF points, `sigma` is the expected measurement noise
(start at 15 pt ~ 5 m, tune empirically).

**Transition probability** (how likely is it to go from symbol s assigned to post i
to symbol t assigned to post i+1):

```
delta = |arcLen(s, t) * scale_K - label_m(i, i+1)|   // meters
p_trans(s -> t) = (1/beta) * exp(-delta / beta)
```

where `arcLen(s, t)` is the arc-length along Cabo Projetado between the two cable-snapped
positions of s and t, `scale_K` is the PDF-to-metre scale, and `beta` is a tolerance
parameter (start at 5 m, tune empirically).

**Order constraint:** The Viterbi lattice enforces monotone arc-order: symbol t for post i+1
must have arc-position >= symbol s for post i (no backtracking on the cable).

**Complexity:** O(n x k^2) where n = posts on page, k = candidates per post. For n=14,
k=20 (pruned from 144 by cable proximity), this is 14 x 400 = 5 600 operations — trivial.

**Why beam search (N3) underperformed:** Beam search explores only the top-B partial
assignments at each step. When the true assignment has a poor local score at step i but
recovers at i+1 (common with zig-zag label positions), beam search prunes the correct
branch. Viterbi never prunes — it evaluates all O(k^2) transitions and keeps the globally
optimal back-pointer. The key difference: Viterbi's transition term penalizes arc-distance
deviations, while N3's beam search scored on bearing and proximity without a proper
transition model.

**Implementation note for browser JS:** The Viterbi DP is 20-30 lines of plain JS:

```javascript
// states[i][j] = best log-prob ending at symbol j for post i
// back[i][j] = which symbol index k at post i-1 led to j
function viterbiAssign(posts, candidatesPerPost, logEmit, logTrans) {
  const n = posts.length;
  const states = posts.map((_, i) => new Float64Array(candidatesPerPost[i].length).fill(-Infinity));
  const back = posts.map((_, i) => new Int32Array(candidatesPerPost[i].length).fill(-1));

  // Init
  candidatesPerPost[0].forEach((_, j) => { states[0][j] = logEmit(0, j); });

  // Forward
  for (let i = 1; i < n; i++) {
    candidatesPerPost[i].forEach((_, j) => {
      let best = -Infinity, bestK = -1;
      candidatesPerPost[i-1].forEach((_, k) => {
        const score = states[i-1][k] + logTrans(i-1, k, i, j);
        if (score > best) { best = score; bestK = k; }
      });
      states[i][j] = best + logEmit(i, j);
      back[i][j] = bestK;
    });
  }

  // Traceback
  const result = new Int32Array(n);
  result[n-1] = states[n-1].indexOf(Math.max(...states[n-1]));
  for (let i = n-2; i >= 0; i--) result[i] = back[i+1][result[i+1]];
  return result;
}
```

No external dependencies needed.

#### 2.1.2 Hungarian Algorithm (Munkres) for Non-Ordered Assignment

**Source:** munkres-js npm package [CITED: npmjs.com/package/munkres-js]
Also: munkres-algorithm npm package [CITED: npmjs.com/package/munkres-algorithm]

For tap poles (off-route, not on the cable sequence) or per-page cross-check assignment
where order does not apply, the Hungarian algorithm gives the globally optimal bipartite
matching in O(n^3). Both packages are browser-compatible, lightweight, no dependencies.

Use case: if N or M anchors need one-to-one assignment to M or N candidates without a
sequence constraint (e.g., first-pass initialization of arc-length anchors on a new page).

For n=14 posts x k=144 symbols = 14x144 cost matrix, Hungarian runs in microseconds in JS.

**When NOT to use:** Do not use Hungarian without the arc-order constraint for the main
cable posts — the ordering constraint is critical and Hungarian does not enforce it
(it would pick the globally cheapest assignment ignoring sequence order, which can produce
assignments where post 5's symbol is arc-located *before* post 3's symbol on the cable).

#### 2.1.3 Longest Common Subsequence / Sequence Alignment (Framing)

**Source:** Wikipedia — Sequence alignment, Needleman-Wunsch algorithm
[ASSUMED — well-known CS algorithm, not verified for this domain]

The Viterbi approach above is equivalent to sequence alignment between:
- Sequence A: the list of Distancia_Poste label values (d1, d2, ..., d_{n-1}) in meters
- Sequence B: the cumulative arc-lengths of all Poste symbol cable-snap positions

This Needleman-Wunsch framing allows gaps (missing distance labels, tap poles) and
mismatches (label rounding error). The cost function is |arc_gap_B - distance_A| in
metres. Gap penalty prevents assigning two consecutive symbols to the same post.

In practice, the Viterbi formulation is cleaner for this problem since it is directly
applicable once you define emission and transition probabilities.

### 2.2 PDF Coordinate Accuracy

#### 2.2.1 PDF Floating-Point Precision

**Source:** O'Reilly — "The Limits of Numbers in SVG" [CITED: oreillymedia.github.io/Using_SVG/extras/ch08-precision.html]
**Source:** Exprodat — "Extracting Vector Data from PDFs" [CITED: exprodat.com/blog/extracting-vector-data-from-pdfs-4/]

PDFs use single-precision floating point (~7 significant decimal digits). For this project:

- Page width: ~1191 PDF points
- Scale: ~0.36 m/pt (from scale factor computation)
- 1 pt = 0.36 m => 1 float ulp at 1000 pt = 0.0001 pt = 0.000036 m

**Conclusion:** PDF float precision is NOT a limiting factor. At 1191 pt page width, the
representable resolution is ~0.0001 pt = ~0.00004 m. The 5–36 m errors are 5 orders of
magnitude above the precision floor. The errors are algorithmic (mis-assignment, wrong CTM
application), not floating-point noise.

#### 2.2.2 CTM Matrix Application in PDF.js

**Source:** PDF.js GitHub discussion #18410 [CITED: github.com/mozilla/pdf.js/discussions/18410]
**Source:** Apryse documentation on PDF coordinates [CITED: docs.apryse.com/web/guides/coordinates]

The PDF coordinate system has (0,0) at the bottom-left with Y increasing upward, while
canvas/screen has (0,0) at the top-left with Y increasing downward. PDF.js handles the
Y-flip via the Current Transformation Matrix (CTM). Known issues:

1. When the CTM is identity-like (x<10 AND y<10 heuristic), the page is a non-detail page
   (already handled by the project's bad-CTM filter).
2. The operator list path construction operators (m, l, c) apply the CTM for each path.
   When PDF.js parses operator list coordinates, the coordinates are in PDF user space
   (post-CTM application). If the CTM has unexpected scaling or rotation, the extracted
   coordinates will appear rotated or scaled relative to the page viewport.

**Risk for this project:** The viewport boxes (pagedNum/x/y/w/h) are extracted from the
overview page and used to map detail-page coordinate origin. If the overview page has a
non-identity CTM (rotation or non-uniform scale), the viewport box coordinates will be in
a rotated frame not matching the detail page frame. This is the most likely cause of the
2.5 degree rotation residual observed on Joao Born page 3.

**Mitigation already implemented:** N4 (per-page rotation parameter). The research
validates this approach is correct.

#### 2.2.3 Geospatial PDF vs. Engineering Drawing PDF

**Source:** FME Safe Software — "Extracting Geospatial Data from PDFs: Top 3 Challenges"
[CITED: fme.safe.com/blog/2018/08/convert-geospatial-pdf/]

INFOVIAS PDFs are engineering drawings exported from CAD, NOT geospatial PDFs. They do not
contain embedded georeference metadata (no OGC encoding, no Adobe supplement). The only
georeference information is the printed UTM grid, which must be detected visually
(implemented as classifyGridLinesFromOps). This is the correct approach. There is no
shortcut via PDF metadata.

### 2.3 Distance-Constrained Arc-Length Placement

#### 2.3.1 Arc-Length Parameterization — The Canonical N1 Approach

**Source:** SVG spec — `getPointAtLength()` [CITED: developer.mozilla.org/en-US/docs/Web/API/SVGGeometryElement/getPointAtLength]
**Source:** npm — `point-at-length` package [CITED: npmjs.com/package/point-at-length]

Arc-length parameterization of a polyline is the standard approach and is already
available in the codebase (`pointAtArcLength` in `cable-builder.js`). The algorithm is:

1. Precompute cumulative arc-lengths for each vertex: `S[0]=0, S[i] = S[i-1] + dist(v_{i-1}, v_i)`
2. Given target arc-length `s`, binary search for the segment `[v_k, v_{k+1}]` where
   `S[k] <= s < S[k+1]`
3. Interpolate: `point = v_k + (s - S[k]) / (S[k+1] - S[k]) * (v_{k+1} - v_k)`

This is O(n log n) preprocessing, O(log n) per query. For n < 200 cable vertices, even
O(n) linear scan is fast enough.

**The core N1 algorithm for this project:**

```
s_anchor = arcSnap(anchor_post, cable)  // arc-length where post 1 hits the cable
s_i = s_anchor
for each consecutive pair (post_i, post_{i+1}):
  label_m = distancia_label(i, i+1)     // meters from PDF label
  delta_pt = label_m / scale_K          // PDF points to advance
  s_{i+1} = s_i + delta_pt
  position_{i+1} = cable.pointAtArcLength(s_{i+1})
```

**Error budget:** For a route with 5 m average spacing between label reads:
- Label rounding: ±0.05 m (labels rounded to 0.1 m)
- Scale factor uncertainty: ±1% (from grid line detection)
- Cumulative over 14 posts: max drift ~ 14 * 0.05 + 300m * 0.01 = 3.7 m

This is well below the 5 m target, even without any correction.

#### 2.3.2 Kalman Filter for Accumulated Distance Error

**Source:** [ASSUMED — standard signal processing approach, not domain-specific literature found]

For very long routes (Siriu, 85 posts), cumulative arc-length drift may compound across
pages. A 1D Kalman filter can be applied to fuse two sources:

- **Process model:** `s_{i+1} = s_i + label_m / scale_K` (arc-length walk)
- **Measurement:** When a GPS anchor is provided for an intermediate post, it gives a
  direct measurement of `s_anchor_gps = arcSnap(gps_i, cable)`

The filter tracks `s_i` (current arc-length) and its variance, correcting at each anchor
post. This is O(n) time, O(1) space per post.

**For the current problem (Joao Born, 14 posts/page, 1 anchor per page):** Kalman is
overkill. The N1 arc-length walk without Kalman should be sufficient. Reserve Kalman for
Siriu (6-sheet, 85-post stress test).

#### 2.3.3 Cross-Page Arc-Length Continuity

The arc-length walk must continue across page boundaries. The already-implemented
`augmentCrossPageDistances` handles this. The research found no better pattern — this
is the correct approach: chain pages by the cable geometry, not by thumbnail offset math.

### 2.4 Georeferencing Engineering Drawings

#### 2.4.1 Affine vs. Similarity Transform — Which is Correct?

**Source:** ESRI — "Understanding Raster Georeferencing"
[CITED: esri.com/about/newsroom/arcuser/understanding-raster-georeferencing]

| Transform | DOF | Min control pts | Preserves shape | Use when |
|-----------|-----|----------------|-----------------|----------|
| Similarity (Helmert) | 4 | 2 | Yes (uniform scale) | Known-good CAD drawings |
| Affine | 6 | 3 | No (shear OK) | Scanned maps with distortion |
| Polynomial 2nd order | 10 | 6 | No | Optical lens distortion |

**Finding for this project:** The current approach is a 3-DOF (Tx, Ty, scale) per-page
transform, extended to 4-DOF (+ rotation) by N4. This is a **Similarity (Helmert)**
transform, which is the correct choice for CAD-exported PDFs (no lens distortion, no
shear). The pathological shear (~26 deg) seen in the best-fit affine is a diagnostic of
**bad input points**, not a real shear — confirming the mis-assignment root cause.

**The affine N5 approach** adds 2 more DOF (shear) but will fit the noise rather than fix
it when inputs are bad. N5 is theoretically valid if inputs are clean; with mis-assigned
poles it will overfit. This confirms the strategy: fix inputs first (N1/Viterbi), then
N5 may help on clean data.

#### 2.4.2 UTM Grid as Control Points — Accuracy Analysis

**Source:** Natural Resources Canada — UTM Grid Reference
[CITED: natural-resources.canada.ca/maps-tools-publications/maps/topographic-maps/utm-grid-civilian-utm-grid-reference]

The UTM grid printed on INFOVIAS detail pages is at 50 m spacing (confirmed: `computeScaleFactor`
returns scale for grid with 50m grid spacing). With 13 paths on page 3 and 12 on pages 4-5:

- Each intersection gives 1 known control point in UTM space
- 50 m grid => the origin uncertainty from extrapolating grid origin to page origin is
  at most 50 m / (number of grid lines visible) in each direction
- With 4+ visible grid lines per page, origin uncertainty is ~ 50/4 = 12.5 m in the
  worst case direction

**Implication:** Even with perfect input positions, the UTM grid alone limits georeferencing
accuracy to ~12 m unless the user provides an explicit GPS anchor. The user-provided GPS
anchor (post 1) is the critical anchor. The 50 m grid gives scale and rotation, but the
absolute origin comes from the anchor.

With 1 anchor: accuracy = anchor_GPS_error + label_rounding_cumulative + scale_error
With 2 anchors (both ends): accuracy = interpolated, much better for middle posts

#### 2.4.3 SIRGAS-2000 Datum

**Source:** IBGE / OSGeo Brazilian CRS wiki
[CITED: wiki.osgeo.org/wiki/Brazilian_Coordinate_Reference_Systems]

SIRGAS 2000 is the official Brazilian datum since 2015. UTM Zone 22S (EPSG:31982) covers
Santa Catarina (Palhoca is in SC). The datum itself introduces < 0.1 m error vs. WGS84
for the Brazilian territory — this is negligible for a 5 m target. The 65 m offset only
applies when incorrectly converting from SAD69 (old datum) to SIRGAS 2000. The project
uses UTM easting/northing from the PDF grid directly (which is already in SIRGAS-2000),
so no datum conversion is needed. No action required.

## 3. Recommended Approach (Implementation Sketch)

### Priority 1 — Enable N1 (Cable Arc-Length Walk) as Default

**Why:** This is the only approach that eliminates the mis-assignment problem entirely. It
does not depend on which Poste symbol is "nearest" — it places each post at a precise
arc-length position on the cable. With correct distance labels, the theoretical error
is < 2 m end-to-end on a single page.

**What to implement:**
1. Remove the `enableCableArcPlacer` opt-in gate. Make it the default path when
   Cabo Projetado is detected on the page.
2. Use the first post's cable-snap as the arc-length anchor `s_0`.
3. Walk forward: `s_{i+1} = s_i + distLabel(i, i+1) / scale_K`.
4. Tap poles (isOffRouteCablePost): keep their current label-anchor position, do not walk.
5. For missing labels (cross-page seam): fall back to Euclidean post-to-post delta
   (`||pos_i+1 - pos_i||`) as the arc-length estimate (already implemented in
   `augmentCrossPageDistances`).

**Expected result:** Joao Born max error drops from 36 m to < 5 m on page 3 (the main
page), and < 10 m end-to-end (cross-page seams add minor drift that two-anchor mode
corrects).

### Priority 2 — Replace N3 Beam Search with Viterbi-HMM for Symbol Selection

**Why:** N3 (beam search) is needed only if N1 is NOT enabled. If N1 is the default, N3
is the fallback for pages without a Cabo Projetado cable. But N3 is also the right
algorithm to use for the *initial cable snap* (step 2 in N1 above): we need to know which
Poste symbol is the anchor for post 1.

**What to implement:**
For anchor selection (post 1 on each page), replace greedy nearest-Poste with a lightweight
Viterbi where k=5 candidates and n=first 3 posts (short lattice):

```javascript
// candidates[i] = up to K Poste symbols near anchor_i, pruned by cable proximity
// logEmit(i, j) = -dist(anchor_i, candidates[i][j])^2 / (2 * sigma^2)
// logTrans(i, k, i+1, j) = -|arcLen(cand_k, cand_j) * scale - label_m(i, i+1)| / beta
const assignment = viterbiAssign(anchors, candidates, logEmit, logTrans);
```

This replaces the current `assignPostPositionsFromPosteSymbols` for multi-post pages.

**Parameters to tune:**
- `sigma`: 15 PDF points (~5.4 m) — emission noise
- `beta`: 15 m — transition tolerance (lenient; cable geometry is approximate)
- `k` (candidates per post): top-10 by distance to cable segment, max 200 pt radius

### Priority 3 — Hungarian for Tap Pole Assignment

For off-route tap poles, use `munkres-js` or a hand-rolled O(n^3) implementation to
assign tap pole labels to their nearest off-cable Poste symbols. This is a non-ordered
assignment — Hungarian is correct here.

### Priority 4 — (Deferred) Kalman Smoothing for Siriu

After N1 is validated on Joao Born and Luiz Carolino, add a 1D Kalman filter for the
arc-length walk when processing 6-sheet routes (Siriu). This amortizes scale uncertainty
across many posts. Only needed if Siriu shows > 5 m drift without it.

## 5. Key Insights Not in Prior Internal Research

### 5.1 The Viterbi vs. Beam Search Distinction Matters

The prior N3 implementation used beam search (width 8), which prunes branches. Beam search
is an approximation of Viterbi. For this problem, the correct branch may have a poor local
score (noisy label position at post i) that recovers at i+1 (because the arc-length
transition constraint is satisfied). Beam search prunes this branch; Viterbi does not.

**Actionable:** Replace beam search with full Viterbi (O(n * k^2)) for n=14, k<=20 — the
exact computation is trivial. The N3 beam search being "width 8" is only a win for large k;
with k<=20 pruned candidates, full Viterbi is faster to implement correctly.

### 5.2 The Arc-Length Walk Completely Bypasses the Assignment Problem

If N1 (arc-length walk) is used as the PRIMARY position source, the Viterbi assignment
(N3 replacement) is only needed for:
a) The anchor post (post 1) on each page — to identify which Poste symbol represents post 1
b) Pages with no Cabo Projetado cable (unknown if this occurs in other client PDFs)

This means: **N1 should be implemented and enabled first**. Viterbi refinement of N3 is
secondary. The priority order is exactly what the internal research recommended.

### 5.3 Two-Anchor Mode Nearly Solves the Problem on Its Own

Looking at debug_results.txt two-anchor mode: posts 21-34 (pages 4-5) are already < 10 m,
and posts 21-25 are < 5 m with two anchors. The remaining problem is pages 3 (posts 5-14),
which are 25-36 m off. This is a pure mis-assignment problem: if N1 correctly places posts
5-14 on page 3 via arc-length walk, and the second anchor (post 34) constrains page 5, the
remaining errors will be < 5 m.

### 5.4 PDF Scale Factor Derivation from Grid is Sound

The `computeScaleFactor` approach (detecting grid line spacing in PDF points, comparing to
known 50 m UTM spacing) is a standard georeferencing technique. The derived scale of
1.266 m/pt confirmed from the grid corresponds to 39.5 PDF pt per 50 m, which gives a
resolution of 1.27 m/pt for the detail pages. The technique is correct. The remaining
error is not in the scale derivation but in the input positions.
