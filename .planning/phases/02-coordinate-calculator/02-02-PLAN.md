Export function detectRouteTopology(posts, distances, cableSegments):
  1. Sort posts by number ascending.
  2. Identify the main route: the longest contiguous number sequence starting from post #1 (1, 2, 3... up to the first gap in sequential numbering).
  3. Identify branches: remaining posts whose numbers start a new sequence (e.g., if main = 1-15, branch = 16-22). A branch starts when there's a number gap AND the next post (e.g., 16) is spatially close to an existing post on the main route or another branch.
  4. IMPORTANT: distinguish branch gaps from OCR gaps. A number gap is a branch boundary ONLY if the posts are spatially far apart (> 100 PDF points). If spatially close (< 100pt), treat it as an OCR miss (missing post in sequence).
  5. For each branch:
     a. Find the junction post: the existing post (from main route or earlier branch) that is spatially closest to the first post of the new sequence in PDF x,y space (D-07). Use simple Euclidean distance: Math.hypot(a.x - b.x, a.y - b.y).
     b. Record the connection: { from: junctionPost.number, to: branchFirstPost.number, isBranch: true }.
  6. Return { mainRoute: number[], branches: [{ start: number, end: number, junctionPost: number }] }.

Modify calculateCoordinates to use detectRouteTopology:
  1. Calculate main route coordinates linearly (as Plan 02-01).
  2. For each branch, propagate GPS from the junction post:
     - branch post 16 GPS = junction post 7's GPS + bearing(7 to 16) + distance(7 to 16) (D-09)
     - Bearing calculation uses the SAME formula from Plan 02-01: atan2(dx, curr.y - next.y)
     - Subsequent branch posts (17, 18...) chain from 16.
  3. Distance for junction to branch-start: look up in distances[] first; if not found, compute from PDF positions using the scale factor (average meters-per-PDF-point from known distances).
</action>

- Branch posts have correct bearing relative to their junction (not relative to the previous main-route post)

Export function detectGaps(posts, distances, cableSegments):
  1. For each sequential post pair (N, N+1) within the same route/branch, check if a cable polyline from cableSegments passes near both posts.
     Implementation: for each cable segment, check if ANY point in its ops[] array (M or L commands) is within a threshold (50 PDF points) of both postA and postB.
     Do NOT import minDistancePointToPathOps -- implement a simpler inline check to avoid circular dependencies:
       const nearPost = (op, post, threshold) => Math.hypot(op.x - post.x, op.y - post.y) < threshold;
       A cable "connects" two posts if it has at least one op near postA AND at least one op near postB.
  2. If no cable connects the pair AND the pair has a null or missing distance entry in distances[], mark it as a gap.
  3. Return gaps: [{ from: number, to: number }].

Modify calculateCoordinates to handle gaps:
  1. When a gap is detected between posts N and N+1:
     a. Compute scale factor = average(meters / pdfDistance) for all post pairs that DO have distance labels (D-11).
     b. Calculate PDF distance = Math.hypot(N+1.x - N.x, N+1.y - N.y).
     c. Estimate meters = pdfDistance * scaleFactor.
     d. Calculate bearing normally from PDF positions (same formula as main route).
     e. Project GPS using estimated distance.
  2. Posts across gaps still receive lat/lon -- the gap only means "no cable drawn" (D-10).
</action>

- Posts across gaps receive calculated lat/lon (not undefined)

- If ALL distances are null (no known pairs), gaps fall back to PDF-only distance without scaling and a warning is emitted
- Gap detection works without importing from cable-builder.js (no circular dependency risk)
