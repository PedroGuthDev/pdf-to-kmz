<objective>
Move the two expensive ingestion operations — `parseDxfText` (134 MB string parse) and `buildPostIndex` (60k-post rbush bulk-load) — off the main thread into a Web Worker (D-04), with a Node inline fallback (Pattern 2) so the timing gate and unit tests run without a browser. Then add the Node timing gate that proves Palhoca.dxf ingests within the 5 s budget (D-05/D-06/SC-4). The worker round-trips `{ posts, cableEdges, primaryCableEdges, rbushDump, extmin, extmax }` via structured clone (rbush toJSON is clone-safe, Pitfall 3); the rbush dump satisfies DXF-04 (region indexed by GPS bbox, index restorable from dump).

Purpose: Closes DXF-06 (large-DXF ingest within ~5 s via off-thread parsing) and DXF-04 (regions indexed by GPS bbox; lookup restorable from the rbush dump). After this slice, a user uploading the 134 MB Palhoca file does not freeze the tab, and the Node gate guarantees the budget in CI.
Output: `dxf-parse.worker.js`, a `runParse(dxfText)` dispatcher in `region-library.js` replacing the inline parse/index, the esbuild change to emit the worker chunk, the timing gate, and its wiring into `npm run test:gate`.

Note: depends on 06-01 because this plan re-edits `region-library.js::addRegion` (the `runParse` swap must sit AFTER the 06-01 validation guards so the gate measures the full validated path) and because both plans modify the same file. Wave 2.
</objective>
