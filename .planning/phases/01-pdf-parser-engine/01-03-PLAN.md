parsePdf() output contract (from SKELETON.md — what index.html must handle):

Success case:
```js
{
  posts:         [{ number: Number, x: Number, y: Number }],
  distances:     [{ from: Number, to: Number, meters: Number|null }],
  cableSegments: [{ id: Number, ops: PathOp[], startPoint: {x,y}|null, endPoint: {x,y}|null }],
  warnings:      String[],
  layerMap:      { allNames: String[] }
}
```

Error cases:
```js
{ error: 'missing_layers', missing: String[], allNames: String[] }
{ error: 'parse_failed',   message: String,   warnings: String[] }
```

Display requirements (D-14 simple summary, D-15 continuous flow):
- Post count: `result.posts.length`
- Distance count: `result.distances.length`
- Cable segment count: `result.cableSegments.length`
- Warnings: `result.warnings` (array of strings)
- Coordinate form: unlock immediately after successful parse (D-15 — no confirmation gate)

File size guard (RESEARCH.md Security V5):
```js
if (file.size > 50 * 1024 * 1024) {
  // show error, return early — do NOT call file.arrayBuffer()
}
```

ESM import pattern (RESEARCH.md Pattern 1 + 01-PATTERNS.md index.html section):
```html
<script type="module">
import { parsePdf } from './parser/pdf-parser.js';
// workerSrc is set by parser/pdf-parser.js at module load time
</script>
```

File input to ArrayBuffer pattern (01-PATTERNS.md index.html section):
```js
const file = e.target.files[0];
const buf = await file.arrayBuffer();
const result = await parsePdf(buf);
```

  <action>
    Create index.html as a single-file browser app. No external CSS framework — plain HTML with minimal inline styles sufficient for readability.

    Structure:
    - `<title>PDF to KMZ</title>`
    - A section for PDF upload: `<input type="file" id="pdfInput" accept=".pdf">` with a label.
    - A div `id="status"` for status/error messages (hidden until needed).
    - A div `id="summary"` for the parse result summary (hidden until parse succeeds).
    - A section `id="coordForm"` for the coordinate input form placeholder (hidden until parse succeeds, per D-15). This section needs only a heading "Step 2: Enter First Post Coordinates" and placeholder text "Coming in Phase 2" — it will be filled in Phase 2. Show it immediately when parse succeeds (D-15 continuous flow, no confirmation gate).
    - A div `id="warnings"` for parser warnings (hidden until there are warnings to show).

    Script block (`<script type="module">`):
    1. Import parsePdf: `import { parsePdf } from './parser/pdf-parser.js';` — workerSrc is already set by pdf-parser.js at module load.
    2. Add change listener to `#pdfInput`.
    3. In the listener: clear previous status/summary/warnings; show "Parsing..." in status.
    4. File size guard: if `file.size > 50 * 1024 * 1024`, show error "File exceeds 50 MB limit" in status and return early without calling arrayBuffer() or parsePdf.
    5. `const buf = await file.arrayBuffer()`.
    6. `const result = await parsePdf(buf)` — wrap in try/catch; any thrown exception shows in status as "Unexpected error: {e.message}".
    7. Handle result:
       - If `result.error === 'missing_layers'`: show error in status: "Missing PDF layers: {result.missing.join(', ')}. Available layers: {result.allNames.join(', ')}". Do NOT show summary or coordForm.
       - If `result.error === 'parse_failed'`: show error in status: "Parse failed: {result.message}". Show any warnings from result.warnings if present.
       - Success (no result.error): hide status or show "Parse complete."; populate and show summary div with: "Posts found: N", "Distances found: N", "Cable segments found: N". Show coordForm immediately (D-15). If result.warnings.length > 0, show warnings div with each warning as a list item.

    The summary display uses simple DOM manipulation (textContent, innerHTML, style.display). No frameworks.

    Inline style rules needed:
    - `#summary`, `#coordForm`, `#warnings`: `display: none` initially.
    - Status messages: `color: red` for errors, `color: green` for success.

    The coordinate form section content is intentionally minimal — just the heading and "Coming in Phase 2" placeholder text. Phase 2 will replace this content.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User file input -> FileReader | User-selected PDF file; untrusted content |
| ArrayBuffer -> parsePdf() | Untrusted binary data enters parser; parser must not throw to DOM |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Denial of Service | File size — very large PDF | mitigate | Size guard before arrayBuffer(): reject if file.size > 50MB with user-visible message |
| T-03-02 | Denial of Service | parsePdf() exception reaching DOM | mitigate | Outer try/catch in file input handler; show "Unexpected error: {e.message}" and stop |
| T-03-03 | Tampering | PDF embedded JavaScript | accept | pdf.js disables PDF JS execution by default; no additional mitigation needed |
| T-03-04 | Spoofing | Non-PDF file uploaded with .pdf extension | accept | parsePdf() returns parse_failed if file is not valid PDF; error shown to user |
</threat_model>
